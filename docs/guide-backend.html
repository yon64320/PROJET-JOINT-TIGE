1. Vue d'ensemble — comment tout s'assemble

Imagine ton application comme un restaurant :

Le navigateur (Chrome) = le client qui s'assoit au restaurant.
Next.js (le serveur) = le serveur qui prend la commande et l'apporte en cuisine.
Les API Routes = les différents guichets en cuisine (un pour les entrées, un pour les plats...).
Supabase (la base de données) = le frigo / le garde-manger où sont stockés tous les ingrédients.
Les migrations = les plans de construction du garde-manger (étagères, tiroirs, étiquettes).

Quand tu cliques sur un bouton dans l'appli, voici ce qui se passe : navigateur → API route → base de données → réponse → affichage. C'est toujours ce cycle.

2. Supabase — ta base de données dans le cloud

Supabase, c'est un service en ligne qui te donne :

  • Une base de données PostgreSQL (le vrai moteur — comme un Excel surpuissant mais invisible)
  • Un système d'authentification (login, mot de passe, sessions)
  • Du stockage de fichiers (pour les plans PDF des équipements)
  • Des règles de sécurité automatiques (RLS — on en parle plus bas)

Si PostgreSQL est le moteur d'une voiture, Supabase c'est la voiture complète : moteur + tableau de bord + clés + GPS. Tu n'as pas besoin de tout assembler toi-même.

Pour communiquer avec Supabase, l'appli utilise des "clients" — des petits programmes qui savent parler le langage de Supabase. On en a 3 (détaillés plus bas).

3. Les 9 tables — le classeur de ta base

Une table en base de données, c'est exactement comme une feuille Excel : des colonnes (en-têtes) et des lignes (données). La différence ? C'est beaucoup plus strict et structuré — chaque colonne a un type précis (texte, nombre, vrai/faux...).

Chaque arrêt de maintenance que tu crées = 1 ligne dans cette table. C'est le point de départ de tout — les OTs et les brides sont rattachés à un projet.

Les clés étrangères (comme project_id, ot_item_id) sont comme des numéros de dossier écrits sur chaque fiche. Chaque bride (flanges) porte le numéro de son OT parent (ot_item_id), qui lui-même porte le numéro du projet (project_id). Si tu supprimes un projet, tout ce qui est rattaché disparaît automatiquement (grâce à ON DELETE CASCADE).

4. Colonnes magiques (GENERATED)

Certaines colonnes se calculent toutes seules. Tu ne peux jamais les modifier directement — elles se mettent à jour automatiquement quand tu modifies les colonnes sources.

Pattern RETENU — "le terrain prime"

Pour chaque donnée, il y a souvent 2 versions : ce que dit le client (BUTA) et ce que tu relèves sur le terrain (EMIS). La valeur RETENU = "prends le terrain si disponible, sinon prends le client".

-- La base fait ce calcul toute seule, automatiquement :
nb_tiges_retenu = COALESCE(nb_tiges_emis, nb_tiges_buta)

-- En français :
-- "Si nb_tiges_emis existe → prends-le
--  Sinon → prends nb_tiges_buta"

Colonnes concernées : nb_tiges_retenu, matiere_tiges_retenu, matiere_joint_retenu.

Pattern DELTA — "alerte divergence"

Quand le terrain dit "DN = 100" mais le client dit "DN = 150", c'est un problème. La colonne DELTA devient true automatiquement pour alerter.

-- La base calcule automatiquement :
delta_dn = true  QUAND  dn_emis ≠ dn_buta  (et les deux existent)
delta_dn = false QUAND  ils sont égaux ou l'un est vide

Tu ne peux JAMAIS écrire directement dans une colonne GENERATED. Si tu essaies de faire un UPDATE sur delta_dn ou nb_tiges_retenu, PostgreSQL te renverra une erreur. Il faut modifier les colonnes sources (dn_emis, dn_buta, etc.).

5. Les migrations — le journal de construction de ta base

Imagine que tu construis une maison. Les migrations, c'est le carnet de chantier : chaque page décrit une étape de construction (fondations → murs → toit → électricité → peinture). Tu les exécutes dans l'ordre, une par une, et à la fin ta maison est complète.

Concrètement, une migration c'est un fichier SQL numéroté qui dit à la base de données quoi créer ou modifier. Elles sont dans le dossier supabase/migrations/.

Quand est-ce qu'une migration s'exécute ?

Pas automatiquement ! C'est toi qui décides quand les appliquer. Il y a 2 manières :

  1. En local : tu copies le SQL et tu l'exécutes dans l'éditeur SQL de Supabase (dashboard)
  2. Via la CLI Supabase : supabase db push envoie toutes les migrations non encore appliquées

Les migrations s'exécutent une seule fois et dans l'ordre. La migration 005 ne peut pas tourner si la 004 n'est pas déjà appliquée. Supabase garde un registre de ce qui a déjà été fait.

Tes 7 migrations, une par une

1

001 — Fondations

Crée les 4 tables principales (projects, ot_items, flanges, operations_ref, dropdown_lists), les index, et les triggers updated_at.

→
2

002 — Fonction RPC

Crée la fonction merge_extra_column pour modifier les colonnes JSONB de manière sûre (atomique, pas de race condition).

→
3

003 — Template PDF

Ajoute la colonne fiche_rob_template sur projects — config du template des fiches robinetterie.

4

Ajoute la colonne responsable sur flanges — pour assigner quelqu'un à chaque vanne.

→
5

Active Row Level Security + ajoute owner_id sur projects. Chaque utilisateur ne voit que ses projets.

→

Chaque migration est incrémentale — elle ajoute quelque chose sans casser ce qui existe déjà. C'est pour ça qu'on utilise IF NOT EXISTS et ADD COLUMN IF NOT EXISTS : si la migration a déjà été jouée, elle ne plante pas.

6. Les 3 clients Supabase — qui parle à la base ?

L'appli a 3 façons différentes de se connecter à Supabase, selon le contexte :

Client Fichier Utilisé où Droits
Browser client supabase-browser.ts Dans le navigateur (composants React) Droits de l'utilisateur connecté (limité par RLS)
SSR client supabase-ssr.ts Sur le serveur, avec les cookies de l'utilisateur Droits de l'utilisateur connecté (respecte RLS)
Admin client supabase-server.ts Dans les API routes (côté serveur) Accès total — contourne toute sécurité

Browser client = un visiteur avec un badge → il ne peut aller que dans les salles autorisées.
SSR client = un employé avec son badge perso → même droits que le visiteur, mais côté serveur.
Admin client = le directeur avec le passe-partout → il ouvre toutes les portes. Les API routes l'utilisent car elles ont déjà vérifié l'identité de l'utilisateur via le middleware. 

// Browser — dans un composant React
const supabase = createBrowserSupabase();
const { data } = await supabase.from('projects').select('*');

// Admin — dans une API route (après vérification auth)
import { supabaseAdmin } from '@/lib/db/supabase-server';
const { data } = await supabaseAdmin.from('flanges').select('*');

7. Le middleware — le vigile de l'appli

Le fichier src/middleware.ts est un gardien qui s'exécute avant chaque requête. Son boulot :

1

Requête arrive

L'utilisateur clique quelque part ou l'appli fait un appel API

→
2

Route publique ?

Si c'est /login, /auth/, un fichier statique → laisse passer sans vérifier

→
3

Vérifie le token

Lit les cookies, vérifie la session Supabase, rafraîchit le token si expiré

→
4

Décision

Connecté → laisse passer.
Pas connecté → redirige vers /login (ou 401 pour les API)

C'est comme le badge à l'entrée de l'usine. Pas de badge → tu entres pas. Badge expiré → on te le renouvelle automatiquement. Zone publique (accueil) → pas besoin de badge.

8. RLS — la sécurité invisible

RLS = Row Level Security. C'est une protection directement dans la base de données.

Imagine un classeur partagé au bureau. Avec RLS, quand tu ouvres le classeur, tu ne vois que tes propres dossiers. Ton collègue, quand il ouvre le même classeur, ne voit que les siens. C'est le même classeur, mais chacun a sa vue.

Concrètement, voici les règles actives (migration 005) :

TableRègle
projectsTu ne vois/modifies/supprimes que les projets où owner_id = toi
ot_itemsTu ne vois que les OTs dont le project_id appartient à un de tes projets
flangesIdem — via le project_id
field_sessionsTu ne vois que tes propres sessions terrain
bolt_specsLecture seule pour tout le monde (données de référence)

Le client Admin (supabaseAdmin) contourne RLS. C'est pour ça qu'on l'utilise uniquement dans les API routes, après avoir vérifié l'identité de l'utilisateur avec le middleware. Double sécurité.

9. Les API Routes — les guichets de l'appli

Chaque API route est un fichier dans src/app/api/. C'est un "guichet" spécialisé : tu lui envoies une requête (GET, POST, PATCH, DELETE) et il te répond avec des données ou une confirmation.

C'est comme les différents guichets à la poste : un pour envoyer un colis (POST), un pour récupérer un colis (GET), un pour modifier une adresse (PATCH), un pour annuler (DELETE). Chaque guichet a ses règles.

Les méthodes HTTP

MéthodeC'est quoiExemple dans l'appli
GETLire des données"Donne-moi tous les OTs du projet X"
POSTCréer quelque chose"Importe ce fichier Excel"
PATCHModifier un champ"Change le statut de cet OT en TC"
DELETESupprimer"Supprime cette session terrain"

Toutes les API routes de l'appli

Projets

GET /api/projects

Récupère la liste de tes projets. Filtre automatiquement par owner_id (tu ne vois que les tiens). Utilisé par la page d'accueil.

Projets

GET + PUT /api/projects/[id]/fiche-template

GET : récupère le template de fiches robinetterie du projet (ou le template par défaut).
PUT : sauvegarde un nouveau template (quand tu personnalises la fiche PDF).

LUT

GET + PATCH /api/ot-items

GET : récupère tous les OTs d'un projet (avec pagination si > 5000 lignes, mais en pratique on met un .limit(5000)).
PATCH : modifie un champ d'un OT. Vérifie que le champ est dans la whitelist (14 champs autorisés). Les colonnes GENERATED sont bloquées.

// Exemple de PATCH envoyé par le tableur quand tu changes une cellule :
{
  "id": "abc-123",          // quel OT
  "field": "statut",        // quelle colonne
  "value": "TC"             // nouvelle valeur
}
J&T

GET + PATCH /api/flanges

GET : récupère toutes les brides d'un projet, avec un join sur ot_items pour avoir le nom de l'ITEM et l'unité.
PATCH : modifie un champ. 65 champs autorisés dans la whitelist ! Si le champ est dans extra_columns, utilise la fonction RPC merge_extra_column pour une mise à jour atomique.

Robinetterie

GET /api/robinetterie

Récupère uniquement les brides marquées rob=true — c'est la vue filtrée pour le tableur robinetterie.

Robinetterie

POST /api/flanges/pair/auto

Appariement automatique : trouve les ITEMs qui ont exactement 2 brides rob non appariées, et les lie (ADM + REF). Utilise un rob_pair_id commun (UUID) et un rob_side (ADM ou REF).

Robinetterie

POST + DELETE /api/flanges/pair

POST : appariement manuel de 2 brides.
DELETE : désapparier une paire (remet rob_pair_id et rob_side à null).

Import

POST /api/import/detect

Phase 1 de l'import : tu uploades ton fichier Excel, l'API scanne les 15 premières lignes pour trouver la ligne d'en-tête, puis fait un fuzzy match (correspondance approximative) entre les noms de colonnes Excel et les champs de la base.

Retourne : le mapping détecté (vert = sûr, jaune = à vérifier, gris = inconnu) + quelques lignes de preview.

Import

POST /api/import/confirm

Phase 2 de l'import : tu as vérifié le mapping et cliqué "Confirmer". L'API parse tout le fichier avec le mapping validé, insère les données en base, et sauvegarde le template pour les prochains imports.

Gère 4 cas : LUT nouvel import, LUT ré-import, J&T nouvel import, J&T ré-import.

PDF

POST /api/pdf/fiches-rob

Génère un PDF de fiches robinetterie. Récupère les données des brides sélectionnées, construit du HTML, et le rend en PDF via Playwright (un navigateur invisible côté serveur).

Terrain

GET + POST + DELETE /api/terrain/sessions

Gestion des sessions terrain : lister, créer (avec sélection des OTs), supprimer.

Terrain

GET /api/terrain/download

Prépare le package hors-ligne : récupère les OTs, brides, spécifications de boulons, listes déroulantes et plans. Tout est envoyé d'un coup pour être stocké dans IndexedDB (la mini-base du navigateur).

Terrain

POST /api/terrain/sync

Synchronisation retour : quand tu reviens du terrain avec du réseau, l'appli envoie toutes les modifications faites hors-ligne. L'API applique chaque mutation avec détection de conflits (si quelqu'un a modifié la même bride entre-temps).

Règle d'or : le terrain gagne toujours.

Terrain

POST /api/terrain/plans

Upload de plans d'équipement (PDF) vers Supabase Storage.

10. L'import Excel — du fichier à la base

Le fuzzy matching en détail

Le fichier synonyms.ts contient un dictionnaire de correspondances :

// L'en-tête Excel dit "DN RELEVE"
// Le champ en base c'est "dn_emis"
// Le dictionnaire de synonymes sait que c'est pareil :

"dn_emis": ["DN", "DN RELEVE", "DN EMIS", "DIAMETRE NOMINAL"]
"dn_buta": ["DN CLIENT", "DN BUTA", "DN DONNEES BUTA"]
"item":    ["ITEM", "NOM", "REPERE", "TAG", "N° EQUIPEMENT"]

L'algorithme fait 3 passes :

  1. Passe 1 — Exact : "DN EMIS" dans l'Excel = "DN EMIS" dans les synonymes → match direct
  2. Passe 2 — Inclusion : "DIAMETRE NOMINAL RELEVE" contient "DIAMETRE NOMINAL" → match
  3. Passe 3 — Levenshtein : "DN EMIIS" (faute de frappe) est très proche de "DN EMIS" → match avec score < 3

Les templates réutilisables

Au premier import, le système calcule une empreinte (fingerprint) des colonnes de ton Excel. Au prochain import d'un fichier similaire, il reconnaît le format et applique le même mapping instantanément. Plus besoin de tout revérifier !

Les utilitaires de conversion

Le fichier src/lib/db/utils.ts contient des fonctions pour convertir les données Excel (qui sont très "sales") en données propres :

FonctionCe qu'elle faitExemple
getStr(val)Extrait du texte propre" V-301 " → "V-301"
getBool(val)Détecte les booléens"X", "OUI", "TRUE" → true
getNumeric(val)Extrait un nombre, filtre le texte"150" → 150, "CALO" → null
getInteger(val)Extrait un entier arrondi"4.7" → 5

Les colonnes Excel peuvent contenir n'importe quoi (du texte dans une colonne nombre, des espaces, des formules...). Ces fonctions nettoient tout ça avant l'insertion en base. Sans elles, PostgreSQL refuserait les données "sales".

12. Comment fonctionne une requête, concrètement

Prenons un exemple réel : tu modifies le DN d'une bride dans le tableur J&T.

Ce qui se passe, étape par étape :

Etape 1 — Tu changes la cellule

Tu passes de 100 à 150 dans la colonne "DN EMIS" de la bride n°42.

Le composant Univer (le tableur) détecte le changement et appelle une fonction JavaScript.

Etape 2 — Le navigateur envoie un PATCH

// Le navigateur envoie cette requête HTTP :
PATCH /api/flanges

// Avec ce corps (body) en JSON :
{
  "id": "uuid-de-la-bride-42",
  "field": "dn_emis",
  "value": 150
}

Etape 3 — Le middleware intercepte

Avant d'arriver à l'API route, le middleware vérifie tes cookies. Token valide ? → OK, la requête continue. Token expiré ? → Rafraîchit automatiquement. Pas de token ? → Erreur 401.

Etape 4 — L'API route traite

// Dans src/app/api/flanges/route.ts :

// 1. Vérifie que "dn_emis" est dans la whitelist
const EDITABLE = new Set(["dn_emis", "dn_buta", "operation", ...]);
// ✅ "dn_emis" est autorisé

// 2. Vérifie que ce n'est PAS une colonne GENERATED
// ❌ "delta_dn" serait refusé ici

// 3. Envoie à Supabase
await supabaseAdmin
  .from('flanges')
  .update({ dn_emis: 150 })
  .eq('id', 'uuid-de-la-bride-42');

Etape 5 — PostgreSQL exécute

-- Ce qui se passe dans la base :

-- 1. Met à jour dn_emis
UPDATE flanges SET dn_emis = 150 WHERE id = 'uuid-42';

-- 2. AUTOMATIQUEMENT, PostgreSQL recalcule delta_dn :
--    dn_emis (150) ≠ dn_buta (100) → delta_dn = TRUE
--    → L'alerte de divergence s'allume toute seule !

-- 3. Le trigger updated_at met à jour la date de modification

Etape 6 — La réponse revient

// L'API renvoie :
{ "success": true }

// Le tableur met à jour l'affichage.
// La colonne DELTA DN passe en rouge (si affichée).

Tout ce processus prend moins d'une seconde. Le middleware, la whitelist, les colonnes GENERATED, le trigger — tout s'enchaîne automatiquement sans que tu le voies.

Les concepts clés à retenir

ConceptExplication en 1 phrase
TableUne feuille Excel dans la base de données, avec des colonnes typées
MigrationUn fichier SQL numéroté qui ajoute/modifie la structure de la base
API RouteUn guichet spécialisé qui reçoit des requêtes et retourne des données
MiddlewareLe vigile qui vérifie ton identité avant chaque requête
RLSUn filtre invisible : tu ne vois que tes propres données
GENERATEDColonnes auto-calculées (RETENU = terrain, DELTA = alerte divergence)
WhitelistListe des champs qu'on a le droit de modifier (sécurité)
extra_columnsFourre-tout JSONB pour les colonnes Excel non reconnues (zéro perte)
Fuzzy matchCorrespondance approximative entre noms de colonnes Excel et champs DB
PWA / IndexedDBMini-base dans le navigateur pour travailler sans réseau
SyncRenvoi des données terrain vers le serveur (le terrain gagne toujours)
Service WorkerProgramme invisible qui cache les pages pour le mode hors-ligne

En résumé : ton appli c'est un restaurant automatisé. Le client (navigateur) passe commande au serveur (API routes), qui la transmet au frigo géant (Supabase). Le vigile (middleware) vérifie les badges, les étagères sont étiquetées (migrations), et certains ingrédients se préparent tout seuls (colonnes GENERATED). Quand le livreur part en zone sans réseau (terrain), il emporte une glacière (IndexedDB) et ramène les nouveaux ingrédients à son retour (sync).
