## HIGH (P0) — bloquants sécurité ou CI rouge

### HIGH-01 — RPC `SECURITY DEFINER` exécutables sur n'importe quel projet

**Source** : RLS F-001 · **Localisation** : `supabase/migrations/001_schema.sql:43-169`

5 fonctions destructrices (`delete_project_cascade`, `_archive_flanges`, `_archive_ot_items`, `reimport_archive_lut`, `reimport_archive_jt`) sont `SECURITY DEFINER` + `GRANT EXECUTE TO authenticated` **sans** check d'ownership interne. Tout user authentifié peut invoquer `POST /rest/v1/rpc/delete_project_cascade` avec le `p_project_id` d'un autre tenant et **détruire** son projet.

**Patch SQL** (à appliquer dans une nouvelle migration `002_security_fixes.sql`) :

```sql
CREATE OR REPLACE FUNCTION delete_project_cascade(p_project_id UUID)
RETURNS VOID AS $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM projects WHERE id = p_project_id AND owner_id = auth.uid()
  ) THEN
    RAISE EXCEPTION 'Projet introuvable ou accès refusé';
  END IF;
  -- ... corps inchangé
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = public;
```

Idem pour `reimport_archive_lut`, `reimport_archive_jt`. Pour `_archive_flanges` / `_archive_ot_items` (helpers internes), `REVOKE EXECUTE FROM authenticated`.

**Tests** : 3 cas Vitest "user A appelle RPC avec project de B → exception".

### HIGH-02 — `POST /api/terrain/sessions` accepte `projectId` arbitraire

**Source** : RLS F-002 · **Localisation** : `src/app/api/terrain/sessions/route.ts:35-99`

Route utilise `supabaseAdmin` (bypass RLS), force `owner_id = user.id` mais **n'audite pas** le `projectId` ni les `otItemIds` reçus. Combiné à `/api/terrain/download` qui ne re-vérifie pas `project_id` sur les fetches `flanges`/`ot_items`/`equipment_plans`, c'est une **lecture totale** des données J&T/LUT d'autres tenants + exfiltration de plans PDF via signedUrl 1h.

**Patch route** : avant l'INSERT, vérifier `projects.owner_id = user.id` puis `ot_items.project_id = projectId` pour chaque `otItemIds`. Voir RLS F-002 §Patch suggéré pour le code complet.

**Patch défense en profondeur** (durcir la policy même si la route reste sur service-role) :

```sql
DROP POLICY field_sessions_insert ON field_sessions;
CREATE POLICY field_sessions_insert ON field_sessions FOR INSERT
  WITH CHECK (
    owner_id = auth.uid()
    AND project_id IN (SELECT id FROM projects WHERE owner_id = auth.uid())
  );
```

(Ce patch couvre simultanément MED-04 / RLS F-007.)

### HIGH-03 — `POST /api/terrain/sync` accepte `flangeId` non validés

**Source** : RLS F-003 · **Localisation** : `src/app/api/terrain/sync/route.ts:33-135`

Route vérifie `sessionId.owner_id = user.id` mais UPDATE des brides via `.in("id", flangeIds)` sans filtrer `flange.ot_item_id ∈ session_items`. **Mutation arbitraire** des brides cross-tenant : changement `field_status="completed"`, écrasement `dn_emis` qui change le RETENU, pollution `commentaires`.

**Patch** : récupérer `field_session_items.ot_item_id` pour la session, fetcher les `flanges` candidates, **filtrer** les mutations dont `flangeId.ot_item_id` n'est pas dans la session. Voir RLS F-003 §Patch.

### HIGH-05 — `POST /api/import/confirm` : pas de check ownership avant ré-import destructif

**Source** : RLS F-005 · **Localisation** : `src/app/api/import/confirm/route.ts:28-220`

Route reçoit `projectId` du formdata et appelle `reimportLutToDb` / `reimportJtToDb` qui invoquent les RPC `reimport_archive_*` (HIGH-01) sans check préalable. Un attaquant Alice peut **vider** le projet de Bob en uploadant un Excel valide avec son propre cookie de session.

const { data: project } = await supabase

  .eq("owner_id", user.id)
  .single();
if (!project) return NextResponse.json({ error: "Projet introuvable" }, { status: 404 });

Le patch HIGH-01 (check `auth.uid()` dans la RPC) ferme la même brèche en profondeur — appliquer les deux.

### HIGH-06 — `import_templates` : RLS UPDATE ouvert (`USING(true)`)

**Source** : RLS F-006 · **Localisation** : `001_schema.sql:564-568`

Tout user authentifié peut **réécrire** les templates d'un autre user. Risque social engineering : modifier `column_mapping` pour mapper `dn_buta` sur la mauvaise colonne, la victime fait un import "automatique" et corrompt ses données silencieusement.

### HIGH-11 — Couverture J&T avec opérations manquantes (donnée, pas code)

### MED-03 — `hasDelta` ignore le format décimal français `"100,5"`

**Source** : Correctness F-005 · **Localisation** : `src/lib/domain/jt.ts:15`

`Number("100,5") === NaN` → `hasDelta("100.5", "100,5") === false`, alors que les valeurs représentent la même chose (faux négatif), ou `hasDelta("100", "200,5") === false` (vrai écart raté). Patch : `Number(v.replace(",", "."))` avant la comparaison. Test verrouillant déjà présent.

### MED-04 — `computeRetenu("", "BUTA")` retourne `""`

**Source** : Correctness F-006 · **Localisation** : `src/lib/domain/jt.ts:6`

`??` ne traite pas `""` comme nullish. Si l'utilisateur efface une cellule EMIS dans le tableur Univer (`""` au lieu de `null`), il **perd la valeur BUTA de fallback**. Patch défensif : `emis?.trim() || buta`. À arbitrer côté UX.
