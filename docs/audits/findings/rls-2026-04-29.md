## Résumé exécutif

- **Findings critiques** : 5
- **Findings élevés** : 1
- **Findings moyens** : 3
- **Findings informatifs** : 4
- **Verdict global** : **🔴 ROUGE — failles d'isolation cross-tenant exploitables par tout utilisateur authentifié**

### Observation principale

Le modèle de défense est **asymétrique** :

- Les routes "standard" (`createServerSupabase`) sont **correctement protégées** par RLS — toutes les routes auditées propagent l'auth utilisateur via cookies, et la RLS filtre sur `owner_id`/`project_id`.
- Les routes "terrain" (`supabaseAdmin` / service-role) **bypass RLS** par conception, et la sécurité repose entièrement sur les checks `eq("owner_id", user.id)` côté code. **Trois des quatre routes terrain manquent un ou plusieurs de ces checks**, créant des IDORs critiques.
- En complément, les **RPC `SECURITY DEFINER`** (`delete_project_cascade`, `reimport_archive_lut`, `reimport_archive_jt`, `_archive_flanges`, `_archive_ot_items`) sont **GRANTED EXECUTE à `authenticated`** sans aucun check d'ownership dans le corps de la fonction. Tout user authentifié peut les invoquer directement via l'API REST Supabase et détruire les données d'un autre tenant — sans même passer par les routes Next.js.

### Périmètre couvert

- Schéma SQL canonique (`supabase/migrations/001_schema.sql`, 637 lignes) — toutes les tables et toutes les RPC.
- 14 routes API actives (la liste du prompt audit en mentionnait 17 mais `flanges/pair`, `flanges/pair/auto` ont été supprimés et `pair_flanges` retirée du schéma — voir commit `b1691d5` et `db-schema.md` mis à jour).
- Helpers : `src/lib/auth/get-user.ts`, `src/lib/db/supabase-server.ts`, `src/lib/db/supabase-ssr.ts`, `src/lib/api/patch-handler.ts`, `src/lib/db/import-lut.ts`, `src/lib/db/import-jt.ts`, `src/middleware.ts`, `src/sw.ts`, `next.config.ts`.

## Findings détaillés

### F-001 — RPC `SECURITY DEFINER` exécutables sur n'importe quel projet — Sévérité : **Critique**

**Localisation** : `supabase/migrations/001_schema.sql:43-54` (`delete_project_cascade`), `:58-114` (`_archive_flanges`), `:118-140` (`_archive_ot_items`), `:143-157` (`reimport_archive_lut`), `:160-169` (`reimport_archive_jt`), `:629` (`GRANT ALL ON ALL FUNCTIONS … TO authenticated`).

**Description**
Cinq fonctions destructrices sont `SECURITY DEFINER` et n'effectuent **aucune** vérification d'ownership dans leur corps. Combinées au `GRANT ALL ON ALL FUNCTIONS … TO authenticated`, n'importe quel utilisateur authentifié peut les invoquer directement via le endpoint REST `POST /rest/v1/rpc/<fn>` avec son JWT, en passant un `p_project_id` arbitraire.

**Reproduction**

```bash
# Avec un user "Alice" authentifié (token JWT de session) :
curl -X POST 'https://<projet>.supabase.co/rest/v1/rpc/delete_project_cascade' \
  -H "apikey: $ANON_KEY" \
  -H "Authorization: Bearer $ALICE_JWT" \
  -H 'Content-Type: application/json' \
  -d '{"p_project_id":"<UUID du projet de Bob>"}'
# → 200 OK. Le projet de Bob et toutes ses dépendances sont supprimés.
```

Idem pour `reimport_archive_lut` / `reimport_archive_jt` qui archivent puis suppriment toutes les `flanges` (et `ot_items` pour LUT) du projet ciblé.

**Impact**
Tout user authentifié peut **détruire totalement** ou **vider** n'importe quel projet d'un autre tenant. Aucune trace d'audit (les archives passent dans `*_archive` mais sont aussi visibles uniquement à l'attaquant si la RLS sur les archives filtre par `project_id IN projects WHERE owner_id` — et le projet vient d'être supprimé pour `delete_project_cascade`).

**Patch suggéré**

Deux approches combinables — la première est suffisante, la seconde ajoute de la défense en profondeur :

1. **Check d'ownership en début de chaque RPC** :

```sql
CREATE OR REPLACE FUNCTION delete_project_cascade(p_project_id UUID)
RETURNS VOID AS $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM projects WHERE id = p_project_id AND owner_id = auth.uid()
  ) THEN
    RAISE EXCEPTION 'Projet introuvable ou accès refusé';
  END IF;
  -- ... le reste inchangé
END;
$$ LANGUAGE plpgsql SECURITY DEFINER SET search_path = public;
```

Idem pour `reimport_archive_lut`, `reimport_archive_jt`. Pour les helpers internes `_archive_flanges` / `_archive_ot_items`, soit ajouter le même check, soit (préférable, voir 2) restreindre l'EXECUTE.

2. **Restreindre EXECUTE aux callers légitimes** :

```sql
REVOKE EXECUTE ON FUNCTION delete_project_cascade(UUID) FROM anon, authenticated;
REVOKE EXECUTE ON FUNCTION reimport_archive_lut(UUID)  FROM anon, authenticated;
REVOKE EXECUTE ON FUNCTION reimport_archive_jt(UUID)   FROM anon, authenticated;
REVOKE EXECUTE ON FUNCTION _archive_flanges(UUID, TEXT) FROM anon, authenticated;
REVOKE EXECUTE ON FUNCTION _archive_ot_items(UUID, TEXT) FROM anon, authenticated;
GRANT  EXECUTE ON FUNCTION delete_project_cascade(UUID) TO service_role;
GRANT  EXECUTE ON FUNCTION reimport_archive_lut(UUID)  TO service_role;
GRANT  EXECUTE ON FUNCTION reimport_archive_jt(UUID)   TO service_role;
```

**Mais** : le caller `/api/projects` DELETE et `/api/import/confirm` utilisent `createServerSupabase` (rôle `authenticated`), pas `service_role`. Si on REVOKE à `authenticated`, il faut soit basculer ces routes sur `supabaseAdmin` (avec check explicite + audit), soit garder `authenticated` mais avec le check intérieur (1).

**Recommandation** : appliquer (1) dans tous les cas (sécurité par défaut). (2) optionnel.

**Tests à ajouter**

- Vitest/RPC : User A authentifié → `supabase.rpc('delete_project_cascade', { p_project_id: <projet de B> })` doit lever `Projet introuvable ou accès refusé`.
- Idem pour `reimport_archive_lut` et `reimport_archive_jt`.

### F-002 — `POST /api/terrain/sessions` accepte `projectId` et `otItemIds` arbitraires — Sévérité : **Critique**

**Localisation** : `src/app/api/terrain/sessions/route.ts:35-99`.

**Description**
La route crée une `field_sessions` avec `owner_id = user.id` (le code force la valeur correcte) **mais ne vérifie pas** que le `projectId` reçu dans le payload appartient au user. Aucune vérification non plus que les `otItemIds` insérés dans `field_session_items` appartiennent bien au projet du user. Comme le client est `supabaseAdmin` (service-role, bypass RLS), les inserts passent silencieusement.

Cette session "frauduleuse" (owner = Alice, project = Bob, items = OTs de Bob) est ensuite exploitée via `/api/terrain/download` (F-002 chemin d'exploitation) qui ne valide que `field_sessions.owner_id = user.id` et fetch les `flanges` et `ot_items` directement par `.in("ot_item_id", otItemIds)` sans filtre owner (`download/route.ts:50-58`).

**Reproduction**

```bash
# Alice authentifiée. UUIDs de Bob obtenus par énumération ou fuite.
curl -X POST 'https://<host>/api/terrain/sessions' \
  -H "Authorization: Bearer $ALICE_JWT" \
  -H 'Content-Type: application/json' \
  -d '{
    "projectId": "<UUID projet Bob>",
    "name": "pwn",
    "otItemIds": ["<UUID ot_item Bob>", "..."]
  }'
# → 201, session créée. sessionId retourné.

curl 'https://<host>/api/terrain/download?sessionId=<id session>' \
  -H "Authorization: Bearer $ALICE_JWT"
# → 200, retourne otItems + flanges + signedUrls des plans de Bob.
```

**Impact**
Lecture totale des `ot_items` et `flanges` (incluant tous les champs `_emis`, `_buta`, `cell_metadata`, `extra_columns`, données terrain) de tout autre tenant. Plus exfiltration des plans PDF via `signedUrl` valide 1h.

**Patch suggéré**

Dans `terrain/sessions/route.ts` POST, avant l'insert :

```ts
// 1. Vérifier ownership du projet
const { data: project } = await supabase
  .from("projects")
  .select("id")
  .eq("id", projectId)
  .eq("owner_id", user.id)
  .single();
if (!project) {
  return NextResponse.json({ error: "Projet introuvable" }, { status: 404 });
}

// 2. Vérifier que tous les otItemIds appartiennent à ce projet
const { data: validOts } = await supabase
  .from("ot_items")
  .select("id")
  .eq("project_id", projectId)
  .in("id", otItemIds);
const validIds = new Set((validOts ?? []).map((o) => o.id));
if (validIds.size !== otItemIds.length) {
  return NextResponse.json(
    { error: "Certains OTs n'appartiennent pas au projet" },
    { status: 400 },
  );
}
```

Et dans `terrain/download/route.ts`, ajouter un filtre `project_id = session.project_id` sur les fetches `ot_items`, `flanges`, `equipment_plans` (en plus du `.in("ot_item_id", otItemIds)`), comme défense en profondeur.

**Patch DB complémentaire** (couvre aussi F-007) — durcir la policy INSERT de `field_sessions` :

```sql
DROP POLICY field_sessions_insert ON field_sessions;
CREATE POLICY field_sessions_insert ON field_sessions FOR INSERT
  WITH CHECK (
    owner_id = auth.uid()
    AND project_id IN (SELECT id FROM projects WHERE owner_id = auth.uid())
  );
```

> Cette policy ne sera évaluée que si la route bascule sur `createServerSupabase` (RLS active). Tant que la route reste sur `supabaseAdmin`, le check côté code reste obligatoire — la policy est défense en profondeur.

**Tests à ajouter**

- Playwright/Vitest E2E : User A POST `/api/terrain/sessions` avec un projectId de B → 404.
- User A POST avec son projectId mais des otItemIds de B → 400.

### F-003 — `POST /api/terrain/sync` accepte des `flangeId` non validés — Sévérité : **Critique**

**Localisation** : `src/app/api/terrain/sync/route.ts:33-135`.

**Description**
La route vérifie que `sessionId` appartient au user (lignes 49-55) puis fait un fetch et UPDATE des brides via `.in("id", flangeIds)` (lignes 81 et 114) **sans** vérifier que ces brides appartiennent à un projet du user, ni même qu'elles sont rattachées à la session. Comme `supabaseAdmin` bypass RLS, les UPDATE écrivent sur n'importe quelle ligne.

**Reproduction**

```bash
# Alice authentifiée, possède sa propre session (sessionId valide).
curl -X POST 'https://<host>/api/terrain/sync' \
  -H "Authorization: Bearer $ALICE_JWT" \
  -H 'Content-Type: application/json' \
  -d '{
    "sessionId": "<session Alice>",
    "mutations": [
      {"flangeId":"<flange Bob>","field":"commentaires","value":"pwned","timestamp":"2026-04-29T00:00:00Z"}
    ]
  }'
# → 200. La bride de Bob est écrasée.
```

Champs whitelistés (`SYNC_ALLOWED_FIELDS`, lignes 10-31) : 20 colonnes incluant tous les `_emis`, `commentaires`, `field_status`, `echaf_*`. Pas de `dn_buta`/`pn_buta` mais le \_emis prime via le triplet `_retenu` GENERATED → l'écrasement de `dn_emis` change la valeur RETENU côté Bob.

**Impact**
Mutation/corruption arbitraire des brides cross-tenant. Vu que `field_status` peut être passé à `"completed"`, l'attaquant peut aussi marquer comme finies les brides d'un autre projet → impact opérationnel direct.

**Patch suggéré**

Avant la boucle d'UPDATE, valider que toutes les `flangeId` sont rattachées à la session (et indirectement au projet du user) :

```ts
// flangeIds déclaré ligne 80
const { data: validFlanges } = await supabase
  .from("flanges")
  .select("id, ot_item_id")
  .in("id", flangeIds);

// Récupérer les ot_item_id autorisés pour la session
const { data: sessionItems } = await supabase
  .from("field_session_items")
  .select("ot_item_id")
  .eq("session_id", sessionId);

const allowedOtIds = new Set((sessionItems ?? []).map((s) => s.ot_item_id));
const allowedFlangeIds = new Set(
  (validFlanges ?? []).filter((f) => allowedOtIds.has(f.ot_item_id)).map((f) => f.id),
);

// Filtrer les mutations : rejeter celles dont flangeId n'est pas autorisé
const validMutations = validMutations.filter((m) => {
  if (!allowedFlangeIds.has(m.flangeId)) {
    errors.push({ mutation: m, error: "Bride hors session" });
    return false;
  }
  return true;
});
```

**Tests à ajouter**

- Vitest : User A sync avec sessionId à lui mais flangeId de B → erreur "Bride hors session", aucune écriture sur la bride de B.

### F-004 — `POST /api/terrain/plans` : pas de vérif d'ownership + path traversal + pas de limite de taille — Sévérité : **Critique**

**Localisation** : `src/app/api/terrain/plans/route.ts:6-53`.

**Description**
Trois défauts cumulés :

1. **Pas de check d'ownership** : `projectId` et `otItemId` sont reçus dans le formdata et utilisés tels quels (lignes 14-15 et 39-43). Avec service-role, l'INSERT dans `equipment_plans` réussit même si `projectId` appartient à un autre tenant — l'attaquant pollue le bucket Storage et la table avec des PDFs visibles depuis le projet de la victime (la RLS de `equipment_plans` autorise `SELECT` à l'owner du projet, donc victime voit le PDF de l'attaquant dans son projet).
2. **Path traversal** : `storagePath = ${projectId}/${otItemId ?? "general"}/${Date.now()}_${file.name}` (ligne 26). `file.name` n'est pas sanitizé → `../../autre-bucket/foo.pdf` reste théoriquement contraint au bucket `plans` mais peut quand même créer des hiérarchies de dossiers parasites et écraser des chemins existants (`Date.now()` n'est pas une garantie d'unicité sous forte concurrence, et le nom n'est pas regex-validé).
3. **Pas de limite de taille** : aucun `if (file.size > MAX) return 413` (cf. la convention `api-conventions.md` qui exige cette validation pour tout upload).

Comparer avec `/api/import/confirm` qui valide bien `MAX_FILE_SIZE = 50 * 1024 * 1024` (`confirm/route.ts:50-53`).

**Reproduction**

```bash
# Alice authentifiée, projectId de Bob obtenu.
curl -X POST 'https://<host>/api/terrain/plans' \
  -H "Authorization: Bearer $ALICE_JWT" \
  -F "file=@evil.pdf" \
  -F "projectId=<UUID projet Bob>" \
  -F "otItemId=<UUID OT Bob>"
# → 201. equipment_plans contient désormais une ligne pointant vers le PDF d'Alice
#         dans le projet de Bob, visible par Bob via terrain/download.
```

**Impact**

- Pollution de projet (insertion de plans non sollicités, potentiellement avec contenu malveillant ou mensonger — risque opérationnel pour le préparateur).
- DoS Storage par upload répétés de gros fichiers.
- Path traversal permettant de créer une arborescence parasite dans le bucket.

**Patch suggéré**

```ts
// 1. Validation taille + MIME (cf. api-conventions.md)
if (!file || !projectId) { ... }
if (file.size > 50 * 1024 * 1024) {
  return NextResponse.json({ error: "Fichier trop volumineux (max 50 Mo)" }, { status: 413 });
}
if (file.type !== "application/pdf") {
  return NextResponse.json({ error: "Type de fichier non accepté" }, { status: 400 });
}

// 2. Check ownership du projet
const { data: project } = await supabase
  .from("projects").select("id").eq("id", projectId).eq("owner_id", user.id).single();
if (!project) return NextResponse.json({ error: "Projet introuvable" }, { status: 404 });

// 3. Check ownership de l'OT (s'il est fourni)
if (otItemId) {
  const { data: ot } = await supabase
    .from("ot_items").select("id").eq("id", otItemId).eq("project_id", projectId).single();
  if (!ot) return NextResponse.json({ error: "OT introuvable" }, { status: 404 });
}

// 4. Sanitize file name
const safeName = file.name.replace(/[^a-zA-Z0-9._-]/g, "_").slice(0, 100);
const storagePath = `${projectId}/${otItemId ?? "general"}/${Date.now()}_${safeName}`;
```

**Tests à ajouter**

- User A POST `/api/terrain/plans` avec projectId de B → 404.
- POST avec `file.name = "../../../etc/passwd.pdf"` → storage_path nettoyé.
- POST avec un fichier de 100 Mo → 413.

### F-005 — `POST /api/import/confirm` : pas de vérif d'ownership de `projectId` avant ré-import — Sévérité : **Critique**

**Localisation** : `src/app/api/import/confirm/route.ts:28-220`, exploitation via `src/lib/db/import-lut.ts:91-93` et `src/lib/db/import-jt.ts:151-153`.

**Description**
La route reçoit `projectId` dans le formdata (ligne 41) et l'utilise directement dans `reimportLutToDb(supabase, rows, projectId)` (ligne 139) ou `reimportJtToDb(...)` (ligne 192). Ces fonctions appellent immédiatement la RPC `SECURITY DEFINER` `reimport_archive_lut` / `reimport_archive_jt` qui **archive puis vide** toutes les `flanges` (et `ot_items` pour LUT) du projet ciblé, **sans vérification d'ownership** ni dans la fonction (F-001), ni dans la route, ni dans la lib.

Le _ré-import_ qui suit (`INSERT ot_items` lignes 127 d'`import-lut.ts`) est filtré par RLS et échoue silencieusement (les inserts de l'attaquant sont rejetés) — mais l'**archive + delete** déjà appliqué reste en place. La victime perd toutes ses brides (et OTs pour LUT) ; il faut restaurer depuis `*_archive`.

**Reproduction**

```bash
# Alice authentifiée. Cookie de session présent (route standard).
# Excel valide, n'importe quel mapping :
curl -X POST 'https://<host>/api/import/confirm' \
  -H "Cookie: <cookie Alice>" \
  -F "file=@n_importe_quoi_valide.xlsx" \
  -F "confirmedMapping=$VALID_MAPPING_JSON" \
  -F "projectId=<UUID projet Bob>"
# → 200. Le projet de Bob est vidé (flanges + ot_items archivés).
```

**Impact**
Destruction de toutes les données J&T/LUT d'un projet d'un autre tenant. Récupérable via `flanges_archive` / `ot_items_archive` mais nécessite intervention manuelle.

**Patch suggéré**

Au début du POST, après la validation du mapping et avant l'appel `reimport*ToDb`, vérifier l'ownership :

```ts
// (déjà fait pour le projet existant ligne 137 - ajouter le check)
if (projectId) {
  const { data: project } = await supabase
    .from("projects")
    .select("id")
    .eq("id", projectId)
    .eq("owner_id", user.id)
    .single();
  if (!project) {
    return NextResponse.json({ error: "Projet introuvable" }, { status: 404 });
  }
}
```

**Note** : ce patch côté route est nécessaire mais pas suffisant. Le patch DB de F-001 (check `auth.uid()` dans `reimport_archive_*`) ferme la même brèche en profondeur, indépendamment des évolutions futures du code.

**Tests à ajouter**

- Vitest : POST `/api/import/confirm` avec projectId de B sous session A → 404, aucun changement sur les données de B.

### F-006 — `import_templates` : RLS `USING(true)` — write cross-tenant — Sévérité : **Élevée**

**Localisation** : `supabase/migrations/001_schema.sql:564-568`.

**Description**
Les policies sont `SELECT USING (true)`, `INSERT WITH CHECK (true)`, `UPDATE USING (true)`. Aucune notion d'`owner_id`. Le commentaire `db-schema.md` justifie la lecture ouverte ("templates réutilisables via `header_fingerprint`"), mais l'**UPDATE ouvert** permet à tout user authentifié de **réécrire** les templates d'un autre user, y compris pour piéger un import légitime (modifier `column_mapping` pour mapper `dn_buta` sur la mauvaise colonne, par exemple — silencieux et difficile à détecter).

**Impact**
Corruption des templates partagés. Risque social-engineering : un attaquant modifie un template légitime, la victime fait un import se fiant au nom du template, données silencieusement corrompues.

**Patch suggéré**

Deux options selon l'intention métier :

A — **Garder le partage en lecture, restreindre l'écriture à un créateur** (recommandé) :

```sql
ALTER TABLE import_templates ADD COLUMN owner_id UUID REFERENCES auth.users(id);
-- Backfill : set owner_id = creator si traçable, sinon NULL
DROP POLICY import_templates_insert ON import_templates;
DROP POLICY import_templates_update ON import_templates;
CREATE POLICY import_templates_insert ON import_templates FOR INSERT
  WITH CHECK (owner_id = auth.uid());
CREATE POLICY import_templates_update ON import_templates FOR UPDATE
  USING (owner_id = auth.uid());
-- SELECT reste ouvert
```

B — **Si les templates sont vraiment globaux** (préparateurs partagent volontairement), au minimum journaliser les modifications (table d'audit) ou désactiver l'UPDATE :

```sql
DROP POLICY import_templates_update ON import_templates;
-- Pas d'UPDATE possible : seule la création de nouvelles versions est autorisée.
```

**Tests à ajouter**

- User A : POST template → User B : tente UPDATE → refusé.

### F-007 — `field_sessions` policy INSERT ne vérifie pas que `project_id` appartient à l'owner — Sévérité : **Moyenne**

**Localisation** : `supabase/migrations/001_schema.sql:587`.

**Description**
La policy `field_sessions_insert WITH CHECK (owner_id = auth.uid())` est correcte sur le `owner_id`, mais ne contraint pas la cohérence avec `project_id`. Comme la route POST `/api/terrain/sessions` utilise `supabaseAdmin` (RLS bypass), cette policy n'est jamais évaluée en pratique — mais c'est une fragilité de défense en profondeur : si demain une route bascule sur RLS, le bug F-002 reviendrait par d'autres moyens.

**Patch suggéré**

```sql
DROP POLICY field_sessions_insert ON field_sessions;
CREATE POLICY field_sessions_insert ON field_sessions FOR INSERT
  WITH CHECK (
    owner_id = auth.uid()
    AND project_id IN (SELECT id FROM projects WHERE owner_id = auth.uid())
  );
```

Lié à F-002. Voir également si même renforcement à appliquer à `equipment_plans_insert` (déjà OK car la policy filtre via projects).

### F-008 — `merge_extra_column` : INVOKER + whitelist correcte mais pas de check id-vs-owner — Sévérité : **Moyenne**

**Localisation** : `supabase/migrations/001_schema.sql:19-34`.

**Description**
La fonction est `LANGUAGE plpgsql` sans `SECURITY DEFINER` → donc INVOKER, RLS s'applique. Whitelist `('ot_items', 'flanges')` correcte, `format('%I', p_table)` échappe correctement l'identifiant (pas d'injection SQL). L'`UPDATE … WHERE id = $3` interne respecte la RLS du caller, donc l'opération est silencieusement no-op si l'`id` appartient à un autre tenant.

**Risque résiduel** : la fonction ne renvoie pas d'erreur en cas de no-op (pas de `IF FOUND`), donc le caller ne sait pas qu'il a tenté de modifier une ligne hors de son périmètre. `patch-handler.ts:56-59` re-fetch ensuite la ligne via `.eq("id", id)` (RLS-filtered) et `.single()` → `single()` lève une erreur si 0 ligne, donc 500 est renvoyé. Comportement défensif final correct mais signal d'erreur peu lisible.

**Patch suggéré (qualité)** :

```sql
-- Option 1 : faire échouer la fonction si aucune ligne n'a été touchée
EXECUTE format(...) USING p_key, p_value, p_id;
IF NOT FOUND THEN
  RAISE EXCEPTION 'Ligne non trouvée ou accès refusé: %', p_id;
END IF;
```

Ou bien retourner un BOOLEAN et que le caller le check.

### F-009 — `POST /api/pdf/fiches-rob` : pas de check `if (!user) return 401` explicite — Sévérité : **Informative**

**Localisation** : `src/app/api/pdf/fiches-rob/route.ts:11-23`.

**Description**
La route inline `createServerClient` au lieu d'utiliser `createServerSupabase` du helper, et n'appelle pas `supabase.auth.getUser()` pour vérifier la session. La RLS protège quand même les requêtes (un user non auth ne verra rien et la route renverra 404), mais c'est une violation de la convention `api-conventions.md` ("Vérification explicite `user.id` au début de chaque handler"). Le middleware bloque en amont la plupart des cas, mais cohérence ≠ sécurité ; un changement de matcher middleware pourrait exposer ce trou.

**Patch suggéré**

```ts
import { createServerSupabase } from "@/lib/db/supabase-ssr";

export async function POST(request: Request) {
  const supabase = await createServerSupabase();
  const {
    data: { user },
  } = await supabase.auth.getUser();
  if (!user) return NextResponse.json({ error: "Non authentifié" }, { status: 401 });
  // ... reste inchangé
}
```

### F-011 — Bypass `SKIP_AUTH=true` en non-prod — Sévérité : **Informative**

**Localisation** : `src/middleware.ts:29-31`.

**Description**
Le middleware bypasse intégralement l'auth si `NODE_ENV !== "production"` ET `SKIP_AUTH === "true"`. Sécurisé tant que NODE_ENV est correctement positionné, mais si un déploiement préprod ou un environnement temporaire oublie de mettre `production`, bypass total.

**Patch suggéré**

Plus défensif : exiger une whitelist d'IPs ou un host header `localhost` :

```ts
const isLocalhost = request.headers.get("host")?.startsWith("localhost") || ...;
if (process.env.NODE_ENV !== "production" && process.env.SKIP_AUTH === "true" && isLocalhost) {
  return NextResponse.next();
}
```

Ou supprimer le bypass et utiliser un user de test seedé avec session simulée.

### F-012 — `.gitignore` couvre `.env*.local` mais pas `.env` simple — Sévérité : **Informative**

## Findings négatifs (rien à signaler)

- **Injection SQL dans `merge_extra_column`** : la whitelist `IF p_table NOT IN ('ot_items', 'flanges')` rejette tout autre identifiant, et `format('%I', p_table)` échappe correctement même avec un identifiant whitelisté. Aucun input direct dans la chaîne SQL. ✓
- **`SET search_path = public`** : présent sur toutes les RPC `SECURITY DEFINER` et les fonctions PL/pgSQL — pas de risque d'escalation par hijacking de schéma. ✓
- **Exposition service-role côté client** : la clé `SUPABASE_SERVICE_ROLE_KEY` est référencée uniquement dans `src/lib/db/supabase-server.ts` et `src/lib/db/seed-bolt-specs.ts`. Aucun import de `supabase-server` depuis un fichier `"use client"` ou un composant. `next.config.ts` n'expose aucune variable serveur. `src/sw.ts` ne contient aucune référence. ✓
- **GRANTs vs RLS** : les GRANTs `ALL ON ALL TABLES TO anon, authenticated, service_role` posent la "porte ouverte", mais la RLS et l'absence de policies INSERT/UPDATE/DELETE sur les tables de référence (`operations_ref`, `dropdown_lists`, `bolt_specs`) bloquent les écritures. ✓
- **Indexes RLS** : toutes les colonnes utilisées dans les policies (`projects.owner_id`, `ot_items.project_id`, `flanges.project_id`, `field_sessions.owner_id`, `equipment_plans.project_id`) ont un index dédié. ✓
- **Bucket Storage `plans`** : privé, accédé uniquement via `createSignedUrl(path, 3600)`. Durée 1h conforme. ✓ (mais voir F-004 pour l'absence de check d'ownership à l'upload).

## Recommandations transverses

- **Bascule autant que possible des routes terrain sur `createServerSupabase`** : la PWA peut envoyer le cookie de session Supabase comme n'importe quel client web — la pattern `Bearer token + service-role + check manuel` est intrinsèquement plus fragile (oubli d'un check = breach). Le service-role doit rester pour des opérations cross-tenant légitimes (seed, jobs admin), pas pour la majorité du trafic utilisateur.
- **Ajouter une suite de tests d'isolation** : Vitest + 2 utilisateurs Supabase Auth seedés, parcours type "Alice tente d'accéder aux données de Bob". Une dizaine de tests couvrirait F-001 à F-005. À automatiser en CI.
- **Documenter dans `.claude/rules/api-conventions.md`** : sous "Terrain API (routes offline/sync)", ajouter une checklist obligatoire :
  > Toute route terrain DOIT vérifier l'ownership de **chaque ressource** reçue dans le payload (`projectId`, `sessionId`, `flangeId`, `otItemId`), pas seulement de la première. Sinon : RLS bypass + aucun garde-fou.
