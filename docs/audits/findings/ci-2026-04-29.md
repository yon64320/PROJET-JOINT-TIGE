### Workflow CI actuel — analyse

Fichier : `.github/workflows/ci.yml` (26 lignes, 1 job `quality`).

**Forces** :

- Pin Node 22 + cache npm.
- Triggers `push` + `pull_request` sur `main`.
- Enchaîne format → lint → type-check → test → build dans le bon ordre.
- Placeholders `NEXT_PUBLIC_SUPABASE_*` pour le build (correct).

**Faiblesses** :

- Tout dans **un seul job** → pas de parallélisation, pas de cache d'artefact, échec d'un step bloque les suivants même si indépendants.
- Pas de `permissions:` → token avec permissions par défaut (large).
- Pas de `concurrency:` → des push successifs lancent des runs cumulatifs.
- Actions sur tag majeur (`@v4`) → vulnérable au tag-hijacking (cf. [GitHub Actions 2026 Security Roadmap](https://github.blog/news-insights/product-news/whats-coming-to-our-github-actions-2026-security-roadmap/)).
- Pas de timeout par step → un step bloqué consomme indéfiniment.
- Pas d'upload d'artifact (build, coverage, sw.js).
- Pas de matrice OS (le projet tourne sur Windows en local mais le CI ne valide que Linux).
- Pas de `--ignore-scripts` ni audit de la chaîne d'install.

### Configurations actuelles

- `vitest.config.ts` (12 lignes) — `environment: node`, include `src/**/*.test.ts`, alias `@/`. **Pas de section `coverage`**.
- `eslint.config.js` (flat config, `^9.39.4`) — TypeScript via `@typescript-eslint/parser`+`eslint-plugin`, 35 globaux déclarés à la main. ⚠️ **Pas de plugin `react`, ni `react-hooks`, ni `next` plugin** — la flat config est très permissive comparée à `.eslintrc.json` qui étendait `next/core-web-vitals`.

- `.husky/pre-commit` = `npx lint-staged`. Pas de pre-push.
- `.github/dependabot.yml` : **absent**.

## Findings détaillés

### 🔴 CRITIQUE

#### F-CI-001 — `next lint` retiré dans Next 16 (CI cassé)

- **Symptôme** : `npm run lint` → `Invalid project directory provided, no such directory: C:\…\lint`. Cause : à partir de Next 15, `next lint` est déprécié, et **Next 16 le retire**. La CLI interprète `lint` comme positional arg = directory.
- **Impact** : workflow `quality` rouge à chaque push.
- **Fix proposé** : remplacer le script par un appel direct à ESLint (flat config présente).
  ```jsonc
  // package.json — proposition
  "lint": "eslint \"src/**/*.{ts,tsx,js,jsx}\"",
  "lint:fix": "eslint \"src/**/*.{ts,tsx,js,jsx}\" --fix"
  ```
- **À traiter avant tout autre travail CI**.

#### F-CI-002 — 26 fichiers non formatés (Prettier)

- **Symptôme** : `format:check` retourne 26 warnings dans `src/app/api`, `src/components/{spreadsheet,terrain,fiche-rob,import}`, `src/lib/{db,domain,excel,jt-views,offline,validation}`, `src/types`.
- **Cause probable** : commits faits via éditeur sans passer par lint-staged (pas de `git add -p` mode interactif) ou rebase qui replay des commits anciens.
- **Fix proposé** : `npm run format` une fois en local + commit dédié `chore: prettier sweep`. Vérifier ensuite que `lint-staged` couvre bien les patterns (déjà OK : `*.{ts,tsx}`, `*.{json,md,css}`).
- **Renforcement** : ajouter un hook `pre-push` avec `npm run format:check` + `npm run lint` + `npm run type-check` pour bloquer un push qui casserait le CI.

### 🟠 ÉLEVÉ

  - À court terme : étudier le remplacement de `xlsx` par `exceljs` (déjà en deps) pour les chemins d'import non-trustés.

#### F-CI-005 — Pas d'E2E malgré Playwright installé

- **Constat** : `playwright ^1.59.1` dans `dependencies` (sic), pas de `playwright.config.ts`, pas de dossier `e2e/`, `serverExternalPackages: ["playwright"]` dans `next.config.ts`.
- **Anomalie** : Playwright en deps **prod** est un signal qu'un import accidentel s'est glissé (ou que Playwright est utilisé runtime — peu probable). À investiguer : `grep -r "from \"playwright\"" src/`.
- **Fix proposé** :
  1. Déplacer `playwright` en `devDependencies` (à valider après vérif d'usage runtime).
  2. Créer `playwright.config.ts` ([Annexe C](#annexe-c)).
  3. Créer `e2e/` avec 3-5 user journeys (cf. [Plan E2E](#plan-e2e)).
  4. Ajouter `"e2e": "playwright test"` au `package.json`.
  5. Stratégie données : voir [Plan E2E §3](#stratégie-données-e2e).

#### F-CI-006 — Permissions `GITHUB_TOKEN` non restreintes

- **Constat** : pas de bloc `permissions:` dans `ci.yml`. Token a les permissions par défaut du repo (potentiellement larges).
- **Fix** : ajouter en tête du workflow : `permissions: { contents: read }`. Élargir au cas par cas si un job nécessite (ex: `pull-requests: write` pour commenter un coverage).

#### F-CI-007 — Actions externes non SHA-pinnées

- **Constat** : `actions/checkout@v4`, `actions/setup-node@v4`. Tag majeur, vulnérable.
- **Fix** : pin sur SHA full-length avec commentaire de version. Procédure :
  ```bash
  gh api /repos/actions/checkout/git/refs/tags/v4.2.2 --jq '.object.sha'
  gh api /repos/actions/setup-node/git/refs/tags/v4.1.0 --jq '.object.sha'
  ```
  Liste des actions à pinner — cf. [Annexe A](#annexe-a).

#### F-CI-008 — Setup ESLint hybride (legacy + flat)

- **Constat** : coexistence `.eslintrc.json` + `eslint.config.js`. ESLint 9 ignore le legacy en mode flat → **`.eslintrc.json` est mort code**, mais induit en erreur. La flat config actuelle **ne charge pas** les règles `next/core-web-vitals` ni `react-hooks` → on perd les checks Next.js (a11y, image optim, etc.).
- **Fix** : supprimer `.eslintrc.json`, étendre `eslint.config.js` avec le plugin `eslint-config-next` en mode flat (Next 16 fournit `eslint-config-next` en flat-compatible) — cf. [doc officielle](https://nextjs.org/docs/app/api-reference/config/eslint).

#### F-CI-009 — Husky pre-commit léger, pas de pre-push

- **Constat** : `.husky/pre-commit` ne fait que `lint-staged` (Prettier). Aucune validation type-check ni lint avant push. Conséquence : un push peut casser le CI alors que tout passe en local.
- **Fix** : ajouter `.husky/pre-push` exécutant `npm run lint && npm run type-check && npm run test` (rapide : ~12 s combiné).

#### F-CI-010 — Pas de concurrency / pas de timeout par job

- **Constat** : push successifs déclenchent runs cumulatifs. Pas de timeout → coût Actions non borné.
- **Fix** :
  ```yaml
  concurrency:
    group: ci-${{ github.workflow }}-${{ github.ref }}
    cancel-in-progress: true
  ```
  Et `timeout-minutes: 15` par job.

## Workflows proposés

### `ci.yml` étendu (proposition — ne pas appliquer sans validation)

```yaml
name: CI
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

permissions:
  contents: read

concurrency:
  group: ci-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

env:
  NODE_VERSION: "22"

jobs:
  quality:
    name: Lint / Format / Types
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@<SHA> # v4.2.2 — cf. Annexe A
      - uses: actions/setup-node@<SHA> # v4.1.0
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: npm
      - run: npm ci --prefer-offline --no-audit
      - run: npm run format:check
      - run: npm run lint
      - run: npm run type-check

  test:
    name: Unit tests + coverage
    runs-on: ubuntu-latest
    timeout-minutes: 10
    steps:
      - uses: actions/checkout@<SHA>
      - uses: actions/setup-node@<SHA>
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: npm
      - run: npm ci --prefer-offline --no-audit
      - run: npm run coverage
      - uses: actions/upload-artifact@<SHA> # v4.4.3
        if: always()
        with:
          name: coverage
          path: coverage/
          retention-days: 14

  build:
    name: Next build
    needs: [quality, test]
    runs-on: ubuntu-latest
    timeout-minutes: 15
    steps:
      - uses: actions/checkout@<SHA>
      - uses: actions/setup-node@<SHA>
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: npm
      - run: npm ci --prefer-offline --no-audit
      - run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: "https://placeholder.supabase.co"
          NEXT_PUBLIC_SUPABASE_ANON_KEY: "placeholder-key-for-build"
      - uses: actions/upload-artifact@<SHA>
        with:
          name: next-build
          path: |
            .next/
            !.next/cache
          retention-days: 7

  e2e:
    name: Playwright E2E (PR only)
    needs: build
    if: github.event_name == 'pull_request'
    runs-on: ubuntu-latest
    timeout-minutes: 20
    steps:
      - uses: actions/checkout@<SHA>
      - uses: actions/setup-node@<SHA>
        with:
          node-version: ${{ env.NODE_VERSION }}
          cache: npm
      - run: npm ci --prefer-offline --no-audit
      - run: npx playwright install --with-deps chromium
      - run: npm run build
        env:
          NEXT_PUBLIC_SUPABASE_URL: ${{ secrets.E2E_SUPABASE_URL }}
          NEXT_PUBLIC_SUPABASE_ANON_KEY: ${{ secrets.E2E_SUPABASE_ANON_KEY }}
          E2E: "1"
      - run: npm run e2e
      - uses: actions/upload-artifact@<SHA>
        if: always()
        with:
          name: playwright-report
          path: playwright-report/
          retention-days: 14
```

### `security.yml` proposé (audit deps + secrets)

```yaml
name: Security
on:
  schedule:
    - cron: "17 6 * * 1" # lundi 06:17 UTC
  workflow_dispatch:
  pull_request:
    branches: [main]
    paths:
      - "package*.json"
      - ".github/workflows/security.yml"

permissions:
  contents: read
  security-events: write # pour upload SARIF

concurrency:
  group: security-${{ github.ref }}
  cancel-in-progress: true

jobs:
  npm-audit:
    runs-on: ubuntu-latest
    timeout-minutes: 5
    steps:
      - uses: actions/checkout@<SHA>
      - uses: actions/setup-node@<SHA>
        with:
          node-version: "22"
          cache: npm
      - run: npm ci --prefer-offline --no-audit
      - name: Audit prod deps (high+ fails)
        run: npm audit --omit=dev --audit-level=high
      - name: Audit all (report only)
        run: npm audit --json > npm-audit.json || true
      - uses: actions/upload-artifact@<SHA>
        if: always()
        with:
          name: npm-audit-${{ github.run_id }}
          path: npm-audit.json
```

## Branch protection — règles à activer côté GitHub UI

> Settings → Branches → Branch protection rules → Add rule → Branch name pattern : `main`

Cocher dans cet ordre :

- [ ] **Require a pull request before merging**
  - [ ] Require approvals: `0` (mono-dev — laisser à 0, ou activer auto-approve sur soi)
  - [ ] Dismiss stale pull request approvals when new commits are pushed
- [ ] **Require status checks to pass before merging**
  - [ ] Require branches to be up to date before merging
  - Status checks requis :
    - [ ] `quality`
    - [ ] `test`
    - [ ] `build`
    - [ ] `e2e` (une fois écrits)
    - [ ] `npm-audit` (une fois `security.yml` activé)
- [ ] **Require conversation resolution before merging**
- [ ] **Require linear history** (optionnel — facilite `git log`)
- [ ] **Require signed commits** (recommandé 2026 — `git config commit.gpgsign true` + clé GPG/SSH GitHub)
- [ ] **Restrict who can push to matching branches** : aucune exception, même pour l'owner.
- [ ] **Restrict force pushes** : ON (sauf cas exceptionnel à justifier).
- [ ] **Restrict deletions** : ON.
- [ ] **Lock branch** : OFF (sauf gel intentionnel).

> **Note mono-dev** : la discipline PR sur soi-même est inhabituelle mais c'est la seule barrière qui force le CI à tourner avant un merge. Sans elle, les status checks ne sont qu'informatifs.
